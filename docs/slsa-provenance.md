# Verifying TrustLink Build Provenance

TrustLink release artifacts are signed with SLSA provenance attestations generated by GitHub Actions. This allows consumers to cryptographically verify that a given artifact was built from a known commit in this repository by the official CI pipeline — not by an untrusted third party.

## What Is Attested

Every release attaches attestations for:

| Artifact | Description |
|---|---|
| `trustlink.wasm` | Unoptimized Soroban WASM binary |
| `trustlink.optimized.wasm` | Production-optimized WASM binary |
| `trustlink-sbom.cdx.json` | CycloneDX SBOM for the Rust contract |
| `trustlink-sdk-sbom.cdx.json` | CycloneDX SBOM for the TypeScript SDK |

npm packages published as `@trustlink/sdk` carry npm's built-in provenance (visible on the npm registry page) because they are published with `--provenance`.

## Prerequisites

Install the GitHub CLI (v2.49+):

```bash
# macOS
brew install gh

# Linux
sudo apt install gh   # or follow https://cli.github.com/manual/installation
```

Authenticate once:

```bash
gh auth login
```

## Verifying the WASM Binary

Download the artifact from a release and verify:

```bash
# Download the artifact from a specific release tag
gh release download v1.2.3 \
  --repo Haroldwonder/TrustLink \
  --pattern "trustlink.optimized.wasm"

# Verify provenance against the repository
gh attestation verify trustlink.optimized.wasm \
  --repo Haroldwonder/TrustLink
```

A passing verification prints the signer's workflow path and commit SHA, for example:

```
Loaded digest sha256:abc123... for file://trustlink.optimized.wasm
Loaded 1 attestation from GitHub API

The following attestation matched the provided artifact:
  - Repo:         Haroldwonder/TrustLink
  - Predicate:    https://slsa.dev/provenance/v1
  - Workflow:     .github/workflows/publish-release.yml@refs/tags/v1.2.3
  - Commit:       abc123def456...

✓ Verification succeeded
```

If the artifact was tampered with, or was not produced by this repository's workflow, verification fails with a non-zero exit code.

## Verifying Both WASM Artifacts in One Command

```bash
gh release download v1.2.3 \
  --repo Haroldwonder/TrustLink \
  --pattern "trustlink*.wasm"

for f in trustlink*.wasm; do
  echo "=== Verifying $f ==="
  gh attestation verify "$f" --repo Haroldwonder/TrustLink
done
```

## Verifying the SBOM

The SBOM itself is also attested, so you can verify that the dependency manifest was produced by CI and was not substituted:

```bash
gh release download v1.2.3 \
  --repo Haroldwonder/TrustLink \
  --pattern "trustlink-sbom.cdx.json"

gh attestation verify trustlink-sbom.cdx.json \
  --repo Haroldwonder/TrustLink
```

## Verifying npm Provenance

npm packages published with `--provenance` embed a transparency log entry. To inspect it:

```bash
npm view @trustlink/sdk@1.2.3 dist.attestations
```

Or on the npm registry website, open the package page and look for the **Provenance** section, which links to the Sigstore transparency log entry.

## How Attestations Are Generated

The release pipeline runs [`actions/attest-build-provenance`](https://github.com/actions/attest-build-provenance) immediately after each artifact is built. The action:

1. Computes the SHA-256 digest of the artifact.
2. Signs a SLSA provenance statement using GitHub's OIDC token (no long-lived secrets).
3. Records the signed bundle in the repository's attestation store.

The attestation captures the repository, workflow file, ref, and commit SHA, giving consumers a cryptographic chain from artifact → GitHub Actions run → source commit.

## Pinning a Verified Artifact in Your Pipeline

After verifying, you can record the digest for reproducible deployments:

```bash
DIGEST=$(sha256sum trustlink.optimized.wasm | awk '{print $1}')
echo "trustlink.optimized.wasm sha256:$DIGEST"
```

In your own CI, verify before deploying:

```bash
gh attestation verify trustlink.optimized.wasm --repo Haroldwonder/TrustLink
stellar contract deploy --wasm trustlink.optimized.wasm ...
```

## Troubleshooting

**`gh attestation verify` returns "no attestations found"**
- Ensure you are using the artifact downloaded from the GitHub release, not rebuilt locally.
- Check that the release was created after SLSA attestation support was added (v1.0.0+).

**Verification fails with "signature mismatch"**
- The artifact may have been modified after publication. Do not deploy it.
- Re-download from the official GitHub release page.

**npm provenance not visible**
- npm provenance requires the package to have been published with `--provenance` and with `id-token: write` permissions in the workflow. All `@trustlink/sdk` releases from v1.0.0 onward include provenance.
